# РуСи89: Автономный компиляторный пайплайн

## Цель проекта
Исследовательский полигон по полной декомпозиции трансляции подмножества C89 в исполняемый ELF64. Каждый этап трансляции инкапсулирован в отдельный скрипт, обменивается данными через читаемые текстовые и CSV-форматы и может быть заменён, отлажен или запущен независимо. Проект написан на чистом Python без внешних зависимостей.

## Архитектура пайплайна
Компилятор разделён на 11 последовательных этапов. Промежуточные артефакты сохраняются на диск, что позволяет анализировать состояние транслятора на любом шаге.

| Скрипт | Этап | Вход -> Выход | Назначение |
|:---|:---|:---|:---|
| `0.py` | Оркестратор | `lib.c` + `test.c` -> `program.elf` | Автоматизирует запуск всех этапов, управляет временными файлами |
| `1.py` | Препроцессинг | `lib.c`, `test.c` -> `merged.c` | Слияние файлов, удаление `//` и `/* */` с сохранением литералов |
| `2.py` | Лексический анализ | `merged.c` -> `tokens.txt` | Контекстный токенизатор C89 + русские ключевые слова |
| `3.py` | Нормализация | `tokens.txt` -> `tokens_flat.txt` | Преобразование в плоский формат `ТИП ЗНАЧЕНИЕ` |
| `4.py` | Синтаксический анализ | `tokens_flat.txt` -> `*_ast.txt` | Pratt-парсер выражений, рекурсивный спуск, построение AST |
| `5.py` | Генерация ассемблера | `*_ast.txt` -> `output.asm` | Трансляция AST в ассемблер x86-64 (Intel синтаксис, полностью совместимый с NASM) |
| `6.py` | Лексер ассемблера | `output.asm` -> `output_tokens.txt` | Токенизация ассемблерного кода для встроенного конвейера |
| `6_1.py` | Экстракция данных | `output_tokens.txt` -> `data_section.csv` | Выделение байтов из директив секции `.data` |
| `7.py` | Первый проход | `output_tokens.txt` -> `1pass_*.csv` | Сбор меток, расчёт размеров инструкций, определение смещений |
| `8.py` | Layout (Размещение) | `1pass_*.csv` -> `layout_*.csv` | Виртуальная адресация ELF64, выравнивание, привязка меток к абсолютным адресам |
| `9.py` | Второй проход | `layout_*.csv`, `1pass_*.csv` -> `2pass_*.csv` | Кодирование инструкций в машинный код x86-64, разрешение переходов |
| `10.py` | Генерация ELF | `2pass_*.csv`, `layout_*.csv` -> `program.elf` | Сборка заголовков `Elf64_Ehdr`/`Phdr`, упаковка секций, установка точки входа |

## Структура файлов
```
.
├── 0.py          # Оркестратор (запуск всего пайплайна)
├── 1.py - 5.py   # Фронтенд (препроцессинг -> AST -> ASM)
├── 6.py - 10.py  # Бэкенд (ASM-лексер -> 1-й проход -> Layout -> 2-й проход -> ELF)
├── lib.c         # Минимальная runtime-библиотека (sys_write, strlen)
├── test.c        # Тестовый исходник на РуСи89
└── README.md     # Этот файл
```

## Быстрый старт
Для запуска полной сборки достаточно выполнить:
```bash
python 0.py
```
При успешном завершении в директории появится `program.elf`:
```bash
./program.elf
```

### Опции оркестратора (`0.py`)
| Флаг | Описание |
|:---|:---|
| `--clean-first` | Очистить временные файлы перед запуском |
| `--keep-temp`   | Сохранить все промежуточные CSV/AST/логи после сборки |
| `--no-elf`      | Удалить `program.elf` при финальной очистке (для CI/CD) |

## Совместимость с NASM
Генератор ассемблера (`5.py`) изначально проектировался как полностью совместимый с инструментарием NASM. Файл `output.asm` может быть собран стандартной цепочкой системных утилит в качестве альтернативы встроенному конвейеру `6.py`–`10.py`:

```bash
python 0.py  # или вручную запустить этапы 1-5
nasm -f elf64 output.asm -o output.o
ld output.o -o output_std
./output_std
```
Это позволяет верифицировать корректность генерации кода, отлаживать AST или использовать системный линкер для проверки ABI без запуска собственного ассемблера.

## Отладка и интроспекция
Проект спроектирован для полной наблюдаемости:
- **AST:** `tokens_flat_ast.txt` и `tokens_flat_ast_with_tokens.txt` содержат дерево разбора с привязкой к исходным токенам.
- **Лог парсера:** В `4.py` включён `DEBUG = True`. Подробности переходов, снимки состояния и ошибки пишутся в `debug_parser.log`.
- **CSV-трейсинг:** Каждый этап ассемблера сохраняет таблицы в формате CSV. Можно точно отследить, как метка превратилась в смещение, а инструкция — в последовательность байт.
- Запуск `0.py --keep-temp` оставит все промежуточные файлы для ручного анализа.

## Зависимости
- Python 3.6+ (используются только `sys`, `os`, `pathlib`, `re`, `struct`, `csv`, `typing`, `enum`)
- Linux x86-64 (для запуска итогового ELF-бинарника)
- Никаких внешних пакетов (`pip install` не требуется)
- Опционально: `nasm`, `ld` (для альтернативной сборки через стандартный тулчейн)

## Планы развития
- Секция `.bss` и поддержка неинициализированных глобальных переменных
- Оптимизация переходов (короткие `jmp rel8` vs `jmp rel32`)
- Семантический анализ: проверка типов, контроль областей видимости, предупреждения о неиспользуемых переменных
- Поддержка препроцессорных макросов (`#define`, `#include`)
- Автоматическая валидация: запуск `program.elf` через `subprocess` и сравнение `stdout` с эталоном
